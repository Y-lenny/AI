# 首发垂直类目与 MVP 交易方案

## 1. 决策结论（摘要）

| 维度 | 决策 |
|------|------|
| **首发垂直类目** | **「本地可视觉化履约 + 强进度诉求」的到家服务**：首包聚焦 **装修/安装/维修巡检**（单城市试点），与规划中「直播监控进度」高度一致。 |
| **第二曲线（文档预留，不纳入 MVP 范围）** | 远程技术外包（屏幕共享 + 里程碑交付），避免首版类目过散。 |
| **MVP 交易方案** | **平台内「里程碑确认 + 线下付款记录」为主**；同步预留 **托管支付** 接口与数据模型，在取得支付牌照/合作通道后切换，不在首版强依赖资金托管。 |

## 2. 首发类目 rationale

- **可视化**：装修/安装过程天然适合手机直播，契合「进度清晰」痛点。
- **纠纷证据**：直播/录像与里程碑绑定，比纯文字工单更易仲裁。
- **冷启动**：地域集中便于地推与双边密度；比「全品类技能市场」更易建立信任模板。
- **边界**：财务咨询、纯开发等 **非强视觉** 类目在 PRD 中归为 **Phase 2**，用屏幕共享与交付物为主。

## 3. MVP 交易：两种模式对比与选择

### 3.1 平台资金托管

- **优点**：信任强、纠纷可冻结资金、商业叙事完整。
- **缺点**：支付牌照/分账/合规成本高，MVP 周期长。

### 3.2 线下付款 + 平台记录（MVP 采用）

- **含义**：双方在平台完成 **报价、里程碑、验收确认**；实际转账走微信/对公等 **线下**；平台记录 **约定金额、节点、确认时间** 与 **可选凭证截图/单号（脱敏）**。
- **优点**：快速验证撮合与视频履约链路；法务与支付耦合低。
- **风险**：平台无法强制回款；需 **用户协议** 明确平台非担保方，并引导大额走电子合同（见合规文档）。
- **升级路径**：订单表预留 `payment_mode: escrow | offline_record`，接入微信/支付宝托管或持牌机构分账后切换。

## 4. 成功指标（MVP 8–12 周）

- 试点城市内 **有效撮合率**、**首单完成率**、**里程碑按时确认率**。
- **开播率**：已匹配任务中关联至少 1 次进度直播的比例。
- **纠纷率**与平均处理时长（即便无自动仲裁，也需人工记录）。

## 5. 开放问题（迭代中回答）

- 试点城市与首批服务方招募渠道。
- 线下记录场景下 **凭证** 的字段设计与隐私脱敏规则。
