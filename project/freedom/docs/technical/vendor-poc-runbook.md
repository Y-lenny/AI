# RTC + 直播厂商 POC 跑通手册

**目的**：在接入业务前，对候选云厂商验证 **1v1 延迟、直播延迟、云端录制、计价模型、移动端推流** ，并输出可复现步骤与结果表。

**候选（国内常见）**：声网 Agora、腾讯云 TRTC + 直播、阿里云 RTS + 直播、即构 ZEGO 等。下文以「厂商 A/B」占位，实测时替换为具体控制台项目。

---

## 1. 环境准备

| 项 | 说明 |
|----|------|
| 测试 App | 各厂商控制台创建测试应用，获取 AppId、密钥（服务端 Token） |
| 本仓库声网（可选） | 在 `server/.env` 配置 `AGORA_APP_ID`、`AGORA_APP_CERTIFICATE` 后，可调用 `POST /sessions/:id/rtc-token` 做端到端联调（频道名 = 创建 `MediaSession` 时的 `providerRoomId`） |
| 网络 | 同一 Wi‑Fi、4G/5G、弱网模拟（Charles / Network Link Conditioner）各一轮 |
| 端 | iOS/Android 真机各 1；Web（Chrome）2 台 |
| 账号 | 2 个测试用户 UID，用于 1v1 |

---

## 2. 1v1 RTC 测试

### 2.1 指标

- **首帧时间**：进房到首帧渲染（ms）
- **端到端延迟**：对端做「举手拍屏」动作，本端观测延迟（粗略：帧计数或秒表，重复 10 次取中位数）
- **弱网**：限制上行 500kbps / 丢包 3% 时是否可维持音频、视频是否自动降档

### 2.2 步骤（模板）

1. 服务端签发 **24h 内短期 Token**（绑定 channel、uid、角色）。
2. 双端进同一 `channel`。
3. 记录控制台/回调中的进房成功时间与首帧回调时间。
4. 填写结果表 §5。

---

## 3. 直播（单主播 + 观众）测试

### 3.1 模式

- **推流**：移动端 SDK 摄像头推流；或 RTMP（OBS）对照组。
- **拉流**：HLS（延迟约 2–10s）与厂商「超低延迟」方案（若提供）对比。

### 3.2 指标

- **开播成功率**：10 次推流成功次数
- **观众首帧**：点击播放到首帧
- **断流恢复**：飞行模式开关 1 次后是否自动恢复

---

## 4. 云端录制与计价

### 4.1 录制

- 开启 **合流录制** 或 **单流录制**（按厂商文档）。
- 确认：录制文件落 **对象存储** 的延迟、文件格式、是否需单独转码费用。

### 4.2 计价（向商务/控制台拉取）

记录以下 **单价维度**（填入 §5）：

- RTC：音频分钟 / 视频 HD 分钟 / 超清档位
- 直播：推流带宽峰值、转码路数、拉流流量、存储、录制分钟
- 免费额度与阶梯折扣（若有）

## 6. 选型建议（决策用）

- **MVP**：优先 **集成成本最低 + 移动端推流稳定 + 录制链路清晰** 的一家；避免首版多厂商抽象。
- **后续**：通过 **Media Provider 接口**（业务层 `issueToken`, `createLiveRoom`, `recordingWebhook`）封装，便于切换。

---

## 7. 与业务绑定（验收标准）

- [ ] Token 可由服务端用环境变量密钥签发，**从不**下发主密钥到客户端。
- [ ] `room_id` / `live_id` 可与本仓库 `Session` / 任务 ID 关联（见 API 设计）。
- [ ] 录制完成 **Webhook** 可更新 `recording_asset` 状态（若 MVP 未接，预留队列）。
- [ ] 与业务 `MediaSession.audienceScope` 对齐：仅任务参与方拉流 vs 公开链接时的 **Token/防盗链策略**（`public_link` 需更短 TTL 与运营开关）。
